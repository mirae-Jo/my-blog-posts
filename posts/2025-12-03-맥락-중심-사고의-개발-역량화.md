# 맥락 중심 사고 개발 전략

왜 맥락 중심이냐... 그게 내 장점이라서
단위로 쪼개지는 정보를 기억하는데 취약한 대신 흐름을 위주로 기억하는 방식이 자괴감 들 정도로 맘에 안들었었는데 이제는 그냥 받아들이고 저 사고체계를 효율적으로 확장시켜서 활용해봐야겠다.


---


### 1. 맥락 중심으로 전체 구조를 먼저 파악

	•	새로운 프로젝트나 기능을 접할 때, 작은 단위 코드부터 보는 대신 전체 흐름, 데이터의 흐름, 기능 간 연결을 먼저 이해합니다.
	•	예: “이 API는 어디서 호출되고, 어떤 컴포넌트가 결과를 쓰는가?”를 시각화하거나 간단한 그림으로 그려보기.
	•	효과: 반복되는 패턴, 중복 기능, 필수 역할을 빠르게 감지 가능.

### 2. 정보 단위는 ‘의미 단위’로 묶어서 기억

	•	단순히 변수명, 함수명, 옵션 등 나노 단위로 기억하려 하지 않고, ‘왜 필요한지’와 ‘어디서 사용되는지’를 함께 기억합니다.
	•	예: fetchUserData() → “사용자 프로필 화면에서 호출, 서버에서 데이터를 가져와 화면에 렌더링”
	•	효과: 나중에 까먹어도 맥락을 통해 기억을 재구성 가능.

### 3. 주기적 리뷰 대신 맥락 재구성

	•	이전에 본 내용이나 구현 경험을 반복적으로 읽는 대신, 프로젝트 전체 맥락 속에서 ‘이 기능이 어디에 연결되는지’를 다시 생각하는 방식으로 리뷰.
	•	효과: 기억이 점점 연결망으로 강화되어 단순 반복보다 효율적.

### 4. ‘중복·불필요’ 체크 포인트 만들기
	
	•	구현하거나 수정할 때, 항상 두 질문을 던집니다:
        1. 이 기능/코드는 이미 다른 곳에서 처리되고 있는가?
        2. 이 코드가 전체 흐름에서 꼭 필요한가?
    •	효과: 서버·클라이언트 중복 요청처럼 불필요한 작업을 초기에 걸러낼 수 있음.

### 5. 속도와 깊이 균형

	•	처음부터 완벽하게 구현하거나 빠르게 하려 하지 않습니다.
	•	대신 맥락 이해 → 핵심 기능 구현 → 세부 최적화 순서로 단계화.
	•	효과: 느린 것처럼 보여도 실제로는 깊이 있는 이해와 재사용 가능한 설계를 확보.

### 6. 시각화 도구 활용

	•	기능 흐름, 데이터 흐름, 컴포넌트 연결을 그림, 다이어그램, 카드 등으로 정리.
	•	시각적 정보는 ADHD 특성이 있는 사람에게 맥락을 빠르게 이해하고 기억을 강화하는 데 도움.
